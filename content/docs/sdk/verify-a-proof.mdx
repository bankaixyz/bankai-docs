`verify_batch_proof` is the single entry point for verification. Pass
it a `ProofBundle` from the SDK, and it walks the full trust chain --
Bankai block proof, MMR header proofs, MPT data proofs -- returning
only data that has been cryptographically verified against the
[stateless light client](../concepts/stateless-light-clients.mdx). The
results are fully trustless. No RPC trust, no external oracle, no
intermediary.

The verifier also preserves request identity in the output. Verified
accounts, storage slots, transactions, and receipts carry the block
they came from plus the address or transaction hash you asked for.

<Callout title="zkVM-compatible out of the box">
`bankai-verify` is `no_std`-compatible. You can run the same
verification logic inside a zkVM, giving your proving system seamless
access to verified on-chain data without any code changes.
</Callout>

## Entry point

```rust
use bankai_types::inputs::ProofBundle;
use bankai_verify::verify_batch_proof;

let results = verify_batch_proof(proof_bundle)?;
```

That single call verifies every proof in the bundle and returns a
`BatchResults` containing only verified outputs. If any proof in the
chain is invalid, the call returns a `VerifyError` and no data is
exposed.

## Verification order

The verifier walks the trust chain in a fixed order:

1. **Bankai block proof** -- verifies the STWO zero-knowledge proof
   and extracts the trusted [Bankai block](../concepts/bankai-blocks.mdx)
   with its committed MMR roots.
2. **Header proofs** -- verifies MMR inclusion proofs for each
   requested header (beacon, execution, or OP Stack) against the
   roots in the trusted block.
3. **Data proofs** -- verifies MPT proofs for accounts, storage slots,
   transactions, and receipts against the state or transaction roots
   in the trusted headers.

Each step depends on the one before it. The block proof produces the
roots that validate headers, and the headers produce the roots that
validate data. Nothing is trusted until its proof checks out.

## Where you can run it

The verifier is a pure function over the proof bundle. It doesn't call
any RPC, doesn't need network access, and doesn't depend on external
state.

- **Server-side** -- verify bundles in your backend before serving
  data to users.
- **Client-side** -- verify bundles directly in the browser or a
  native client, removing any trust in your own backend.
- **Inside a zkVM** -- the crate is `no_std` by default. Disable the
  `std` feature and run the exact same `verify_batch_proof` inside
  a proving system to produce a proof over verified on-chain data.

## Result shape

`verify_batch_proof` returns a `BatchResults` with two top-level
groups:

```rust
pub struct BatchResults {
    pub evm: EvmResults,
    pub op_stack: OpStackResults,
}
```

You only need to read the group that matches the requests you made in
the batch. The top level separates Ethereum-family results from
OP Stack results, and the leaf values carry the exact identity of the
request that produced them.

### Ethereum results

Access verified Ethereum data through `results.evm`:

| Field | Type | What it contains |
|---|---|---|
| `execution_header` | `Vec<ExecutionHeader>` | Verified execution headers (alloy `Header`). |
| `beacon_header` | `Vec<BeaconHeader>` | Verified beacon headers -- slot, proposer index, state root, body root. |
| `account` | `Vec<VerifiedAccount>` | Verified accounts with `block`, `address`, and `account` fields. |
| `storage_slot` | `Vec<VerifiedStorageSlots>` | Verified storage slot groups with `block`, `address`, and `slots` fields. |
| `tx` | `Vec<VerifiedTransaction>` | Verified transactions with `block`, `tx_hash`, `tx_index`, and `tx` fields. |
| `receipt` | `Vec<VerifiedReceipt>` | Verified receipts with `block`, `tx_hash`, `tx_index`, and `receipt` fields. |

### OP Stack results

Access verified OP Stack data through `results.op_stack`. The shape
mirrors the Ethereum side:

| Field | Type | What it contains |
|---|---|---|
| `header` | `Vec<ExecutionHeader>` | Verified OP headers. |
| `account` | `Vec<VerifiedAccount>` | Verified OP accounts with `block`, `address`, and `account` fields. |
| `storage_slot` | `Vec<VerifiedStorageSlots>` | Verified OP storage slot groups with `block`, `address`, and `slots` fields. |
| `tx` | `Vec<VerifiedTransaction>` | Verified OP transactions with `block`, `tx_hash`, `tx_index`, and `tx` fields. |
| `receipt` | `Vec<VerifiedReceipt>` | Verified OP receipts with `block`, `tx_hash`, `tx_index`, and `receipt` fields. |

## Understanding the result shape

`BatchResults` is intentionally shallow at the top and precise at the
leaves. Headers stay close to their native types, but accounts,
storage slots, transactions, and receipts are wrapped so the verified
value always carries its provenance with it.

That provenance matters as soon as you ask for the same address at two
different blocks, or the same transaction shape across multiple
networks. You should not have to reconstruct "what this value belongs
to" from request order alone.

### The shared identity wrapper

Every wrapped result starts with the same block reference:

```rust
pub struct BlockRef {
    pub network_id: u64,
    pub block_number: u64,
}
```

`network_id` is the numeric chain ID for the verified chain state, and
`block_number` is the exact block height that state was verified
against. For Ethereum execution results, that is the L1 execution
block. For OP Stack results, that is the OP chain block.

### Account and storage results

Account-like results keep both the block reference and the address you
requested:

```rust
pub struct VerifiedAccount {
    pub block: BlockRef,
    pub address: Address,
    pub account: TrieAccount,
}

pub struct VerifiedStorageSlots {
    pub block: BlockRef,
    pub address: Address,
    pub slots: Vec<(U256, U256)>,
}
```

Use these wrappers when you need both the verified payload and the
identity of the account it came from.

| Field | Meaning |
|---|---|
| `block` | The verified chain block that anchors the account or storage view. |
| `address` | The account address you asked the SDK to prove. |
| `account` | The verified account payload -- nonce, balance, storage root, and code hash. |
| `slots` | The verified `(slot_key, slot_value)` pairs for that account and block. |

### Transaction and receipt results

Transaction-like results preserve both the transaction hash and the
resolved transaction index inside the block:

```rust
pub struct VerifiedTransaction {
    pub block: BlockRef,
    pub tx_hash: FixedBytes<32>,
    pub tx_index: u64,
    pub tx: TxEnvelope,
}

pub struct VerifiedReceipt {
    pub block: BlockRef,
    pub tx_hash: FixedBytes<32>,
    pub tx_index: u64,
    pub receipt: ReceiptEnvelope,
}
```

This gives you enough information to display or persist a verified
transaction or receipt without separately tracking which block it came
from.

| Field | Meaning |
|---|---|
| `block` | The verified chain block that contains the transaction. |
| `tx_hash` | The transaction hash you requested. |
| `tx_index` | The transaction's index in that block. |
| `tx` | The verified transaction payload. |
| `receipt` | The verified receipt payload. |

### Why headers are different

Headers are returned as raw header types because the header itself
already carries the main identity fields you usually need, such as
block number, roots, and hashes. The wrappers exist for result types
whose raw payload would otherwise lose important request context.

These wrappers matter when you verify the same address or transaction
across multiple blocks or networks. You don't need to recover
provenance from request order -- each verified value carries its own
identity.

### Accessing results

Results are returned in the same order you requested them in the
batch builder. A typical access pattern:

```rust
let results = verify_batch_proof(proof_bundle)?;

let account = &results.evm.account[0];
println!(
    "Account {} has balance {} at block {}",
    account.address,
    account.account.balance,
    account.block.block_number
);

let storage = &results.evm.storage_slot[0];
for (key, value) in &storage.slots {
    println!("Slot {} = {}", key, value);
}
```
